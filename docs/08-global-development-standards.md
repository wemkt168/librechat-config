# 全局开发规范模板

> **重要**：这是全局开发规范模板，适用于所有项目。  
> 每个新项目都应该参考这个模板，创建自己的项目宪章。

---

## 📋 使用方式

### 对于新项目

1. **复制模板**：将本文件复制到新项目的 `docs/` 目录
2. **创建项目宪章**：参考 `07-project-charter.md` 的格式，创建项目特定的宪章
3. **定制规范**：根据项目特点，定制 Zeabur 项目结构、服务组织方式等

### 对于 AI 助手

**自动读取规则**：
- 进入任何新项目时，优先检查是否存在 `docs/07-project-charter.md` 或 `docs/00-project-charter.md`
- 如果存在，自动读取并遵循项目宪章
- 如果不存在，提醒用户创建项目宪章

## 🏗️ 通用项目组织原则

### 1. 代码组织

- **所有代码必须在版本控制中**（GitHub/GitLab 等）
- **禁止在部署平台直接编辑代码**（如 Zeabur、Vercel 等）
- **每个服务独立目录**，包含完整的代码和文档

### 2. 文档规范

- **每个服务必须有 README.md**，说明功能、部署位置、环境变量
- **项目必须有项目总览文档**（`00-project-overview.md`）
- **项目必须有项目宪章**（`07-project-charter.md`），明确项目结构和服务组织方式

### 3. 部署规范

- **明确部署位置**：哪个平台、哪个项目、哪个服务
- **环境变量管理**：所有敏感信息通过环境变量配置，不在代码中硬编码
- **服务独立性**：每个服务可以独立部署、更新、重启

### 4. 命名规范

- **服务目录**：小写字母和连字符（如 `video-proxy`、`image-proxy`）
- **服务名称**：与目录名称一致
- **文件命名**：小写字母和连字符

## 📝 项目宪章模板结构

每个项目的宪章应该包含：

1. **文档位置与作用范围**
   - 明确这是项目特定的宪章
   - 说明适用的项目名称和仓库

2. **部署平台项目结构**
   - 列出所有相关项目（如 Zeabur Projects）
   - 说明每个项目的用途和服务列表
   - 明确服务与项目的映射关系

3. **GitHub 仓库结构**
   - 代码组织方式
   - 服务目录结构

4. **服务部署映射关系**
   - GitHub 仓库 → 部署平台的映射
   - 工作目录设置
   - 环境变量配置

5. **代码组织规范**
   - 服务目录结构要求
   - README.md 必须包含的内容
   - 命名规范

6. **禁止事项**
   - 防止代码四散
   - 防止项目混乱
   - 防止硬编码

7. **新增服务检查清单**
   - 创建新服务时必须完成的步骤

## 🔄 持续改进机制

### 如何完善宪章

1. **遇到问题时记录**：
   - 在项目宪章中添加"常见问题"或"注意事项"
   - 记录容易出错的地方

2. **成功经验总结**：
   - 记录最佳实践
   - 记录有效的组织方式

3. **定期审查**：
   - 每个季度审查一次项目宪章
   - 根据项目发展更新规范

4. **跨项目学习**：
   - 从其他项目的宪章中学习好的做法
   - 将通用经验更新到本模板
